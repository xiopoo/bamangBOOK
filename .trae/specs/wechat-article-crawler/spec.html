微信公众号历史文章抓取脚本 - 产品需求文档

Overview

  • Summary: 设计并实现一个自动化爬虫脚本，用于系统性地抓取指定微信公众号发布的全部历史文章数据，包括文章元数据（标题、发布时间、链接、阅读量、点赞数等）和文章内容（正文文本、图片资源、作者信息等），并以结构化格式存储，支持增量更新、频率控制、错误重试等生产级特性。
  • Purpose: 为内容运营、数据分析、知识沉淀等场景提供高效、稳定、合规的微信公众号历史数据采集能力，解决手动收集效率低、易遗漏、格式不统一等问题。
  • Target Users: 内容运营人员、数据分析师、知识管理工作者、研究人员。

Goals

  • 完整抓取指定公众号的全部历史文章列表，无遗漏
  • 抓取每篇文章的完整内容（正文、图片、作者等）
  • 结构化存储（JSON + CSV + 本地文件），支持增量更新
  • 合理的请求频率控制，避免对服务器造成过度负载
  • 完善的错误处理和重试机制，保证抓取稳定性
  • 灵活的配置选项，满足不同抓取需求
  • 严格遵守法律法规和平台协议，确保合规性

Non-Goals (Out of Scope)

  • 不绕过微信公众平台的登录认证机制
  • 不抓取需要付费或权限验证才能访问的内容
  • 不提供微信公众号管理后台的操作功能（如发布、回复等）
  • 不实现分布式爬虫架构
  • 不提供 GUI 图形界面（仅命令行 / 配置文件方式）
  • 不对抓取内容进行 NLP 分析或智能处理（仅采集和存储）
  • 不支持视频号、小程序等非文章类型内容

Background & Context

  • 微信公众号是国内最大的内容平台之一，大量优质内容以公众号文章形式存在
  • 手动收集和整理公众号历史文章效率极低，且容易遗漏
  • 微信公众平台未提供公开的历史文章批量导出 API
  • 现有开源工具普遍存在功能不完备、稳定性差、频率控制不当等问题
  • 合规性是爬虫项目的核心前提，必须严格遵守《网络安全法》、《数据安全法》及平台规则
  • 技术选型：Python 3.10+，利用 requests + BeautifulSoup4 作为核心抓取库

Functional Requirements

FR-1: 公众号历史文章列表抓取

  • 支持通过公众号名称或微信号（biz参数）定位目标公众号
  • 自动获取公众号的全部历史文章列表
  • 每条记录包含：文章标题、发布时间、文章链接（永久链接）、摘要、封面图、阅读量、点赞数、在看数、评论数等关键信息
  • 能够遍历所有分页内容，确保无遗漏

FR-2: 文章内容深度抓取

  • 对列表中的每篇文章进行详情页抓取
  • 提取正文文本内容，保留段落结构
  • 下载文章中的所有图片资源，保存到本地目录
  • 提取作者信息、原创标识、文章分类标签
  • 记录文章的阅读量、点赞数、在看数等互动数据

FR-3: 分页遍历与去重

  • 自动识别并遍历所有分页，直至抓取全部历史文章
  • 基于文章 URL 或文章 ID 进行去重判断
  • 记录抓取进度，支持断点续抓

FR-4: 结构化数据存储

  • 支持多种存储格式：JSON（完整结构化数据）、CSV（表格数据，便于 Excel 打开）
  • 图片资源按文章 ID 分目录存储
  • 元数据与内容分离存储，便于检索和增量更新
  • 提供统一的数据 schema，字段定义清晰

FR-5: 增量更新机制

  • 记录上次抓取时间戳，仅抓取新增或更新的文章
  • 支持按时间范围筛选（开始日期 / 结束日期）
  • 对已抓取的文章可选择跳过或覆盖更新
  • 增量模式下自动对比已有数据，避免重复抓取

FR-6: 请求频率控制

  • 可配置的请求间隔（最小间隔、最大间隔）
  • 支持随机延迟，模拟人工浏览行为
  • 支持单篇文章抓取后延迟、分页切换后延迟等多级频率控制
  • 提供「温和」「标准」「快速」三档预设频率

FR-7: 错误处理与重试机制

  • 网络超时自动重试，可配置重试次数
  • HTTP 错误状态码（429、500、502、503 等）识别与重试
  • 指数退避重试策略，避免加剧服务器压力
  • 部分文章抓取失败不影响整体流程，失败记录单独日志
  • 抓取过程中断后可恢复（断点续抓）

FR-8: 配置化管理

  • 支持通过配置文件（YAML / JSON）指定所有参数
  • 支持命令行参数覆盖配置文件
  • 关键配置项包括：目标公众号名称/ID、数据存储路径、抓取时间范围、频率控制参数、重试参数、代理设置等
  • 配置文件模板提供默认值和注释说明

FR-9: 日志与进度监控

  • 详细的运行日志（INFO / WARNING / ERROR 三级）
  • 实时显示抓取进度（总数 / 已完成 / 失败数 / 预计剩余时间）
  • 抓取完成后输出统计报告（成功数、失败数、总耗时、平均速度等）
  • 失败文章列表单独导出，便于人工复核

FR-10: 合规与道德约束

  • 内置 robots.txt 检查逻辑，遵守平台爬取规则
  • 明确的 User-Agent 标识，便于平台识别和联系
  • 抓取内容仅限个人学习或研究用途，不得用于商业用途
  • 不存储或传播敏感个人信息
  • 代码注释和文档中明确说明合规使用要求

Non-Functional Requirements

  • NFR-1（性能）: 标准模式下，单篇文章平均抓取时间 ≤ 5 秒（含延迟）；1000 篇文章完整抓取 ≤ 2 小时
  • NFR-2（稳定性）: 网络波动情况下，整体抓取成功率 ≥ 95%；单次运行中断后可在 1 分钟内恢复继续
  • NFR-3（可维护性）: 代码模块化设计，核心模块（抓取、解析、存储、调度）分离；关键逻辑有单元测试覆盖
  • NFR-4（可移植性）: 支持 macOS / Linux / Windows 跨平台运行；仅依赖 Python 标准库和主流第三方库
  • NFR-5（安全性）: 不存储用户登录凭证；所有网络请求均为 HTTPS；配置文件中的敏感信息（如 Cookie）支持环境变量注入
  • NFR-6（合规性）: 严格遵守 robots.txt 协议；请求频率在合理范围内；代码和文档中包含明确的合规声明

Constraints

  • Technical:

    • - Python 3.10+
    - 核心依赖：requests、beautifulsoup4、lxml、pandas、PyYAML
    - 不使用 Selenium / Playwright 等重型浏览器自动化工具（降低复杂度和资源占用）

  • Business:

    • - 仅供个人学习和研究使用，不得用于商业目的
    - 不得用于大规模批量爬取或对目标平台造成压力
    - 用户需自行承担使用本工具的法律责任

  • Dependencies:

    • - 依赖微信公众平台网页端的页面结构（可能随平台更新而变化，需维护）
    - 依赖用户提供的有效访问凭证（Cookie / Token）
    - 图片下载依赖稳定的网络连接

Assumptions

  • 用户已拥有合法的微信公众号访问权限（即能正常浏览公众号文章）
  • 微信公众平台的页面结构在短期内不会发生重大变化
  • 用户能够提供必要的访问凭证（如 Cookie）
  • 抓取的数据用于个人学习、研究或内容存档等合法用途
  • 网络环境能够正常访问微信公众平台

Acceptance Criteria

AC-1: 文章列表完整抓取

  • Given: 用户配置了有效的公众号标识和访问凭证
  • When: 运行脚本执行全量抓取
  • Then: 抓取到的文章总数与公众号实际发布数量一致（误差 ≤ 1%），每篇文章都包含标题、发布时间、链接、阅读量、点赞数等关键字段
  • Verification: programmatic
  • Notes: 通过随机抽样人工核对数量和字段完整性

AC-2: 分页遍历无遗漏

  • Given: 目标公众号历史文章超过 100 篇（跨越多页）
  • When: 脚本完成全部抓取
  • Then: 所有分页的文章均被抓取，无分页遗漏，文章 ID 去重正确
  • Verification: programmatic
  • Notes: 检查最早一篇和最晚一篇文章是否都在结果中

AC-3: 文章内容抓取完整

  • Given: 脚本已完成文章列表抓取
  • When: 对随机抽取的 10 篇文章进行内容详情抓取
  • Then: 每篇文章的正文文本完整（与网页版对比，段落数一致），图片资源全部下载到本地（数量与网页一致），作者信息正确
  • Verification: human-judgment
  • Notes: 人工对比网页原文与抓取结果

AC-4: 结构化存储格式正确

  • Given: 抓取完成
  • When: 检查输出目录
  • Then: JSON 文件格式合法、字段完整；CSV 文件可用 Excel 正常打开且无乱码；图片文件按目录分类且均可正常打开
  • Verification: programmatic

AC-5: 增量更新有效

  • Given: 已完成一次全量抓取，且之后公众号有新文章发布
  • When: 以增量模式再次运行脚本
  • Then: 仅抓取新增的文章，已有文章不重复抓取；抓取完成后总文章数 = 原有数量 + 新增数量
  • Verification: programmatic

AC-6: 请求频率控制生效

  • Given: 配置了请求间隔参数（如最小 2 秒，最大 5 秒）
  • When: 运行脚本并观察请求时间间隔
  • Then: 相邻两次请求的间隔在配置范围内，无短时间内大量集中请求的情况
  • Verification: programmatic

AC-7: 错误重试机制有效

  • Given: 模拟网络超时或 5xx 错误场景
  • When: 脚本遇到请求失败
  • Then: 自动按指数退避策略重试，重试次数不超过配置值；重试成功后继续后续任务；重试失败则记录错误日志并继续下一篇
  • Verification: programmatic

AC-8: 配置选项完整可用

  • Given: 用户提供配置文件或命令行参数
  • When: 运行脚本
  • Then: 所有配置项（公众号、存储路径、时间范围、频率、重试、代理等）均生效，行为与配置一致
  • Verification: programmatic

AC-9: 断点续抓功能正常

  • Given: 抓取过程中手动中断脚本
  • When: 再次运行脚本
  • Then: 从上次中断位置继续，已成功抓取的文章不重复抓取
  • Verification: programmatic

AC-10: 合规性说明清晰

  • Given: 项目根目录和代码中
  • When: 查看 README 和代码注释
  • Then: 包含明确的合规声明、使用限制、法律责任提示；User-Agent 中包含可识别的标识和联系方式
  • Verification: human-judgment
