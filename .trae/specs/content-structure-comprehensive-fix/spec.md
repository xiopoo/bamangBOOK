# 内容结构全面修复（标题层级 + 纯文本表格）- 产品需求文档

## Overview
- **Summary**: 系统性修复文档中的标题层级混乱问题和大量未识别的纯文本表格问题，确保目录结构逻辑连贯、所有表格数据以标准 Markdown 格式正确呈现。
- **Purpose**: 解决因 OCR/转录导致的内容结构问题——标题层级倒置（h3 下出现 h2）、列表项被误设为标题、业务板块标题截断、大量数字数据以纯文本形式呈现而非表格——从而提升内容的可读性、专业性和导航体验。
- **Target Users**: 深度阅读财务内容的投资者、研究员，以及依赖目录导航的所有读者。

## Goals
- 修复所有 P0 标题层级问题（h3 下出现 h2、列表项误设为标题、标题截断）
- 修复股东信和合伙人信中的主要纯文本表格（投资组合表、财务数据表、保险业务表、年度对比表）
- 确保目录导航逻辑连贯，标题层级符合 h1 > h2 > h3 > h4 的规范
- 保持原始数据的完整性和准确性，不修改任何数字或文字内容

## Non-Goals (Out of Scope)
- 不修复 articles/ 等其他目录的表格（聚焦 letters/ 和 partnership/）
- 不涉及样式层面的修改（表格样式已在前几轮标准化）
- 不重写或润色正文内容
- 不处理低置信度的疑似表格（避免误判）
- 不修复所有 299 处表格（分批处理，本批聚焦最常见、影响最大的类型）

## Background & Context
- 前几轮已修复：表格重复表头、列数不匹配、尾部空列、简单表格纯文本化、股东信标题提取、财务报表（损益表/资产负债表/收益来源表）表格化
- 本轮发现更深层的结构问题：
  1. **标题层级混乱**：h3 小标题下出现 h2 大标题，导致目录结构倒置
  2. **列表项误设为标题**：如"2，将普通股面值减少到1美分。"被设为 h2
  3. **业务板块标题截断**：如"内布拉斯加家具"应为"内布拉斯加家具店"
  4. **投资组合表纯文本化**：141 处持仓数据表以纯文本形式存在
  5. **财务年度对比表纯文本化**：58 处多年份财务数据对比表以纯文本形式存在

## Functional Requirements
- **FR-1**: 修复标题层级倒置问题（h3 下出现 h2 的情况）
- **FR-2**: 修复列表项/正文句子被误设为标题的问题
- **FR-3**: 修复业务板块标题截断问题（如"内布拉斯加家具"→"内布拉斯加家具店"）
- **FR-4**: 修复投资组合表（股份数量、公司名称、成本、市值等）为标准 Markdown 表格
- **FR-5**: 修复保险业务数据表（浮存金、承保损失、综合比率等）为标准 Markdown 表格
- **FR-6**: 修复财务年度对比表（利润表、资产负债表多年对比）为标准 Markdown 表格
- **FR-7**: 确保修复后的标题前后有空行，符合 Markdown 规范

## Non-Functional Requirements
- **NFR-1**: 数据准确性——所有表格数字、文字与原文完全一致
- **NFR-2**: 层级正确性——标题层级符合 h1 > h2 > h3 规范，目录逻辑连贯
- **NFR-3**: 可读性——表格在桌面端和移动端均能正常显示
- **NFR-4**: 可回滚性——所有修改可通过 Git 历史追溯和回滚

## Constraints
- **Technical**: 基于现有 Markdown 渲染管线，不引入新依赖
- **Business**: 内容必须忠实于原文，不得随意增删
- **Scope**: 本批处理 letters/ 和 partnership/ 目录下的 P0 问题

## Assumptions
- 标题层级倒置问题中，应将 h3 提升为 h2，还是将后续 h2 降为 h3？—— 需根据内容判断：如果 h3 是章节主题（如"透视收益"），应提升为 h2；如果 h2 是子板块，应降为 h3
- 投资组合表的列结构相对固定，可以批量处理后逐一核对
- 保险业务表的结构在不同年份之间有延续性，可以参考已修复的年份

## Acceptance Criteria

### AC-1: 标题层级倒置问题修复
- **Given**: 股东信中存在 h3 标题下直接出现 h2 标题的情况
- **When**: 修复完成后
- **Then**: 标题层级符合 h1 > h2 > h3 规范，目录结构逻辑连贯
- **Verification**: `human-judgment`
- **Notes**: 需根据内容上下文判断是提升 h3 还是降低 h2

### AC-2: 列表项误设标题修复
- **Given**: 列表项（如"2，将普通股面值减少到1美分。"）被设为 h2/h3 标题
- **When**: 修复完成后
- **Then**: 列表项恢复为正常正文，不再出现在目录中
- **Verification**: `human-judgment`

### AC-3: 业务板块标题截断修复
- **Given**: "内布拉斯加家具"等不完整的业务板块标题
- **When**: 修复完成后
- **Then**: 标题完整（如"内布拉斯加家具店"），与正文衔接自然
- **Verification**: `human-judgment`

### AC-4: 投资组合表表格化
- **Given**: 纯文本形式的投资组合持仓表
- **When**: 转换为标准 Markdown 表格后
- **Then**: 表格包含正确的列（股份数量、公司名称、成本、市值等），数据完整对齐
- **Verification**: `human-judgment`

### AC-5: 保险业务表表格化
- **Given**: 纯文本形式的保险业务数据表（浮存金、承保损失等）
- **When**: 转换为标准 Markdown 表格后
- **Then**: 表格包含年度和各项指标列，数据完整准确
- **Verification**: `human-judgment`

### AC-6: 财务年度对比表表格化
- **Given**: 纯文本形式的多年份财务数据对比表
- **When**: 转换为标准 Markdown 表格后
- **Then**: 表格包含各年度列和财务项目行，数据完整准确
- **Verification**: `human-judgment`

### AC-7: 浏览器端正确渲染
- **Given**: 修复后的内容页面
- **When**: 在浏览器中访问
- **Then**: 标题层级正确显示，表格以表格形式正确渲染
- **Verification**: `human-judgment`

## Open Questions
- [ ] 标题层级倒置时，是提升上层 h3 还是降低下层 h2？（建议：根据内容主题判断，主章节用 h2，子板块用 h3）
- [ ] 299 处表格数量较大，本批优先处理哪些类型？（建议：投资组合表 + 保险业务表 + 年度对比表，约 238 处）
- [ ] 合伙人信中的列表项误设标题是否也需要修复？（建议：是，合伙人信也在导航范围内）
